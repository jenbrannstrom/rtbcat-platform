# Incident & Fixes Handover — 2026-03-27

## Summary

scan.rtb.cat was down with "Application error" in the browser. Root cause was **disk 100% full** on the production VM. This triggered a cascade of issues that were resolved in one session, plus several pre-existing bugs were discovered and fixed.

---

## 1. Disk Full (Root Cause of Outage)

**What happened:** The 80GB production disk (`catscan-production-sg`) hit 100%. A legacy 29GB SQLite database (`catscan.db`, last modified Feb 7) was never cleaned up after the Postgres migration. Combined with 26GB of imported CSVs and 8GB of parquet files, there was zero free space.

**Impact:**
- Docker health checks failed (`no space left on device` writing temp files)
- Next.js dashboard couldn't write to temp/cache, causing client-side React exceptions
- Gmail imports stopped on March 22 (couldn't write files)
- nginx access logging stopped on March 23

**Resolution:**
- Deleted `catscan.db` (29GB) — confirmed no writes since Feb 7, all data in Postgres
- Deleted 2,177 imported CSVs (26GB) — confirmed all data imported into Postgres (138M rows in rtb_daily)
- Resized disk from 80GB → 150GB (~$6/mo more)
- Grew filesystem online (no downtime)
- Result: 121GB free, 17% used

**Outstanding:**
- Gmail imports resumed and are catching up (4-day gap: March 22–26)
- CSVs will accumulate again from new imports (~430MB/day). Consider a cron job to auto-delete CSVs after import, or archive to GCS. Gmail keeps the attachments, GCS large files have 30-day retention.

---

## 2. AppsFlyer Postback Endpoint Blocked by Auth

**What happened:** The `/conversions/appsflyer/postback` endpoint was not in the session auth `PUBLIC_PATHS`. Server-to-server postbacks from AppsFlyer were rejected with "Authentication required".

**Resolution:** Added all conversion postback endpoints to `PUBLIC_PREFIXES` in both `api/session_middleware.py` and `api/auth.py`. These endpoints have their own webhook secret/HMAC verification.

**Commit:** `552ba806`

**Outstanding:** No AppsFlyer traffic has been received yet. The client may need to re-enable or re-test the postback URL in their AppsFlyer dashboard. The endpoint is confirmed working (returns "Invalid or missing webhook secret" for requests without the proper secret).

---

## 3. Video URLs Broken in Creative Modals

**What happened:** VAST XML video URL extraction regex had double-escaped brackets (`\\[` instead of `\[`), causing it to never match `<![CDATA[...]]>` wrappers. Every video creative showed "No video URL available."

**This was a pre-existing bug** — never worked since the regex was written. Creatives with a direct `videoUrl` field in raw_data were unaffected, but most video creatives only have VAST XML.

**Resolution:** Fixed regex in `services/creative_preview_service.py` line 104.

**Commit:** `e2bc177b`

---

## 4. Creatives Page: Spend Sort Was Client-Side Only

**What happened:** The "Spend" sort button only sorted the 120 already-loaded creatives in the browser. The API returned creatives sorted by `created_at`, so the first page showed low-spend creatives while top spenders ($500+) were buried on later pages.

**Resolution:**
- Added `sort_by` parameter to the `/creatives` and `/creatives/v2` API endpoints
- When `sort_by=spend`, the store queries `rtb_daily` for top creative IDs by spend, then fetches those creatives — two fast queries instead of one slow JOIN
- Dashboard passes `sort_by=spend` when the sort button is active
- Button renamed to "Spend Hi/Lo" / "Default Order"

**Commits:** `9744c395`, `3182e5ae`, `d9ba1d13`

---

## 5. Batch Performance Endpoint Used Raw Table Instead of Precompute

**What happened:** The `/performance/metrics/batch` endpoint queried `rtb_daily` (138M rows) directly for batch creative performance, taking 41+ seconds for 30-day queries. The precomputed `config_creative_daily` table (177K rows, 0.35s) existed but wasn't being used.

**This is a design violation.** The precompute system is a core architectural decision. The batch endpoint was written with `rtb_daily` as its source and never switched to precompute as the dataset grew.

**Resolution:** Rewrote `get_creative_summaries()` in `creative_performance_repo.py` to use `config_creative_daily` as primary source. Falls back to `rtb_daily` only for creatives missing from the precomputed table.

**Trade-off:** `config_creative_daily` does not have clicks data. CTR and CPC are unavailable in batch list views. Single-creative detail views still use `rtb_daily` (fast for 1 ID) and have full metrics.

**Commit:** `01a93614`

**Prevention:** Added `CLAUDE.md` with "precompute-first" rule to prevent future endpoints from querying `rtb_daily` in batch.

---

## 6. Database Index Added

Created `idx_rtb_daily_buyer_date_creative_spend` on `rtb_daily (buyer_account_id, metric_date, creative_id, spend_micros)` — used by the spend sort query. Build completed successfully.

Attempted a covering index on `(creative_id, metric_date) INCLUDE (spend_micros, impressions, clicks)` but it was OOM-killed (4GB VM RAM insufficient for sorting 138M rows). **Not needed** since we switched to precomputed tables.
