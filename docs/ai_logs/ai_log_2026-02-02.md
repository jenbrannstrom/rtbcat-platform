**VM2 CLOUD SQL PROXY - ARCHITECTURE DECISION**

**Problem:** Cloud SQL Auth Proxy must bind to `127.0.0.1` (security requirement), but Docker containers connect via `host.docker.internal` which resolves to `172.17.0.1` (Docker bridge). Containers couldn't reach the proxy.

**Decision:** Run Cloud SQL Proxy as a Docker service instead of systemd.

**Implementation (commits 2e6e8b6, 19c79fb):**

```yaml
# docker-compose.gcp.yml
services:
  cloudsql-proxy:
    image: gcr.io/cloud-sql-connectors/cloud-sql-proxy:2.14.3
    container_name: catscan-cloudsql-proxy
    command:
      - "--address"
      - "0.0.0.0"
      - "--port"
      - "5432"
      - "${CLOUDSQL_INSTANCE:-catscan-prod-202601:asia-southeast1:catscan-production-serving}"
    restart: always

  api:
    depends_on:
      - cloudsql-proxy
    # ...
```

**Key decisions:**
1. Proxy binds to `0.0.0.0` inside container network (safe - not exposed to host)
2. API connects via Docker DNS: `cloudsql-proxy:5432`
3. GCE VMs use metadata server for auth (no credentials file needed)
4. Systemd `cloud-sql-proxy.service` disabled on VM2

**.env change:**
```
# Before (didn't work with 127.0.0.1 proxy)
POSTGRES_DSN=...@host.docker.internal:5432/...

# After (uses Docker service name)
POSTGRES_DSN=...@cloudsql-proxy:5432/...
```

## ARCHITECTURE NOTES

### Why Docker proxy instead of systemd?

1. **Network isolation:** Systemd proxy on `127.0.0.1` unreachable from containers
2. **Security:** Binding systemd proxy to `0.0.0.0` exposes it to external network
3. **Docker solution:** Proxy in same Docker network, `0.0.0.0` only exposed internally
4. **Simplicity:** Single `docker compose up -d` starts everything

### GCE Metadata Authentication

On GCE VMs, cloud-sql-proxy authenticates via the instance's attached service account:
- No `GOOGLE_APPLICATION_CREDENTIALS` file needed
- Proxy reaches `169.254.169.254` metadata server automatically
- Service account: `catscan-production-sa@catscan-prod-202601.iam.gserviceaccount.com`

### Orphan Container Issue

When mixing systemd catscan service with manual `docker compose`, orphan containers appeared (e.g., `b3d902eab7e6_catscan-cloudsql-proxy`).

**Fix:** `docker compose down --remove-orphans` before `up -d`

**Prevention:** Keep systemd catscan service disabled until updated to use same compose file.
