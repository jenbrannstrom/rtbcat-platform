# AI Log — 2026-02-06

## VM2 Nginx Auth Popup Fix + Login/Registration Notes

### Problem
Nginx had `auth_basic` enabled on both `/` and `/api`, which caused the browser's native HTTP Basic Auth prompt to appear before the app's own login page could load.

### Fix (VM2)
Updated `/etc/nginx/sites-enabled/catscan` on VM2:
- Removed `auth_basic` from `/` and `/api` so the app handles auth via its login page and session cookies.
- Added a dedicated `/api/auth/` location block so login/register/callback endpoints are publicly accessible.
- Kept OAuth2 Proxy endpoints (`/oauth2/`) available for Google SSO.
- Backed up old config to `/etc/nginx/sites-available/catscan.bak.2026-02-06`.

**Result:** The login page at `https://vm2.scan.rtb.cat/login` loads without the HTTP Basic Auth popup.

## How Login Works Now (VM2)

### Primary login path (Google OAuth via OAuth2 Proxy)
1. Go to `https://vm2.scan.rtb.cat/login`.
2. Click **Sign in with Google**.
3. OAuth2 Proxy authenticates you and forwards `X-Email` to the API.
4. The backend auto-creates the user on first login. The **first user becomes admin**; subsequent users become `user`.

### Other login options (if enabled)
- **Authing OIDC:** The login page includes Authing; works only if Authing env vars are configured.
- **Email/password:** The login page includes password login. It is **disabled for general signup** after the first user unless an admin explicitly registers users.

---

## Who Can Log In?

1. **Google OAuth users allowed by OAuth2 Proxy**
   - Controlled by the OAuth2 Proxy config in `/etc/oauth2-proxy.cfg`.
   - If `email_domains` is set, only those domains can sign in.
   - If `authenticated_emails_file` is used, only listed emails can sign in.

2. **Users created in the app database**
   - Any Google user who is allowed by OAuth2 Proxy will be auto-created on first login.
   - The **first user** is `admin`; later users are `user`.

---

## Registration (Email/Password)

Registration is **not open** after the first user:
- The **first user** can register with email/password and becomes `admin`.
- After that, **only an admin** can register new users (via admin UI or API).

Relevant backend behavior:
- `POST /auth/register`
  - If there are **no users**, it allows registration and auto-logs in the first user (admin).
  - If users exist, it requires the requester to already be an **admin**.

---

## Can I Limit Registration / Access?

Yes, in multiple layers:

1. **OAuth2 Proxy allowlist** (recommended):
   - Set `email_domains = ["company.com"]` in `/etc/oauth2-proxy.cfg` to restrict by domain.
   - Or use `authenticated_emails_file` to allow only specific email addresses.
   - Restart OAuth2 Proxy after changes:
     ```bash
     sudo systemctl restart oauth2-proxy
     sudo systemctl restart nginx
     ```

2. **Admin user management** (app level):
   - Admins can deactivate users from the admin UI or via `/admin/users` endpoints.

3. **Registration control (password auth)**:
   - `/auth/register` already blocks new registrations unless the requester is an admin.
