## Root Cause + Fix: Gmail Import Reliability ("SSH pipe break")

Problem raised:
- Operational runs were reporting `exit code 255` from background SSH/IAP sessions.
- Import often completed, but execution model was fragile and looked unreliable.

Root cause:
- `255` reflected SSH transport disconnects, not always import failure.
- More importantly, scheduled/manual imports were launched via in-process FastAPI
  background tasks. Those are tied to API worker lifecycle and can be interrupted
  by restarts/deploy churn.

Implemented fix:
- Move import execution to a detached worker process, launched from API, independent
  of request/SSH connection lifecycle.

Commit:
- `30824e0` — Run Gmail imports in detached worker process

Files changed:
- `services/gmail_service.py`
  - removed in-process `BackgroundTasks` import execution
  - added `_spawn_import_worker(job_id)` using `subprocess.Popen(..., start_new_session=True)`
- `api/routers/gmail.py`
  - updated `/gmail/import` and `/gmail/import/scheduled` to call new queue method
- `scripts/gmail_import_worker.py` (new)
  - runs `run_import(...)`
  - on successful imports, triggers:
    - `refresh_home_summaries(days=30)`
    - `refresh_config_breakdowns(days=30)`
  - logs execution lifecycle and exit state

Operational effect:
- Scheduler/manual trigger returns quickly (`queued=true`).
- Worker continues independently of SSH/API request connection stability.
- Worker logs are written to:
  - `~/.catscan/logs/gmail_import_worker.log`
- Status remains observable via:
  - `/gmail/status`

Validation:
- Python compile checks passed:
  - `api/routers/gmail.py`
  - `services/gmail_service.py`
  - `scripts/gmail_import_worker.py`
