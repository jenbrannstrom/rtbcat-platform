## VM2 Login Loop - Root Cause Analysis

### Symptom
`https://vm2.scan.rtb.cat/` redirects to `/oauth2/sign_in` in an infinite loop after successful Google OAuth authentication.

### Investigation Timeline

1. **cookie_secure = true applied** - Set in `/etc/oauth2-proxy.cfg` and restarted oauth2-proxy. Verified on line 8 of config. Cookie startup log confirmed: `secure(https):true`. Loop persisted.

2. **nginx config reviewed** - HTTPS via Let's Encrypt is correctly configured. `auth_request /oauth2/auth` + `error_page 401 = /oauth2/sign_in` pattern is correct. No issues found in nginx layer.

3. **oauth2-proxy logs analyzed** - Key finding: OAuth flow completes successfully. Google authenticates `cat-scan@rtb.cat`, callback returns 302, and the next `/oauth2/auth` checks return **202** (cookie present, user authenticated). Cookie IS working at the oauth2-proxy layer.

4. **nginx access log revealed the real flow:**
   ```
   01:15:50  /oauth2/callback      -> 302 (auth succeeds, cookie set)
   01:15:51  /                     -> 304 (page loads, auth passes)
   01:15:51  /api/auth/check       -> 200 (35 bytes)  <-- Dashboard API call
   01:15:52  /oauth2/sign_in       -> 200              <-- JS redirect!
   ```
   The dashboard loads successfully, then makes a client-side `fetch("/api/auth/check")` call. The 35-byte response is `{"authenticated":false,"user":null}`.

5. **Frontend auth flow** (`dashboard/src/contexts/auth-context.tsx`):
   - `AuthProvider` mounts -> calls `fetch("/api/auth/check")`
   - If `data.authenticated` is false -> `setUser(null)`
   - After init, if `!user` -> `window.location.href = "/oauth2/sign_in"` (line 129) -> **LOOP**

6. **Backend auth flow** (`api/session_middleware.py` + `api/auth_oauth_proxy.py`):
   - `/auth/check` is a PUBLIC_PATH (line 39) - skips mandatory auth
   - For public paths, middleware only attaches user if `X-Email` header present AND `is_oauth2_proxy_enabled()` returns true
   - `OAUTH2_PROXY_ENABLED=true` IS set in the Docker container (confirmed via `docker exec`)
   - User lookup calls `_get_or_create_oauth2_user(email)` which queries PostgreSQL

### Root Cause

7. **Database unreachable** - Direct curl test with X-Email header:
   ```
   $ curl -s -H 'X-Email: cat-scan@rtb.cat' http://127.0.0.1:8000/auth/check
   {"authenticated":false,"user":null}
   ```
   API container log:
   ```
   ERROR - Failed to get/create OAuth2 user: failed to resolve host 'cloudsql-proxy': [Errno -3] Temporary failure in name resolution
   ```

8. **Cloud SQL Proxy container is dead:**
   ```
   85f9af342fa9  cloud-sql-proxy:2.14.3  Exited (143) 13 hours ago
   ```
   The API and dashboard containers were recreated 13 hours ago, but the Cloud SQL Proxy was not restarted. Without the database, `_get_or_create_oauth2_user()` catches the exception and returns `None`, so `request.state.user` is never set.

### Root Cause Chain
```
Cloud SQL Proxy container down
  -> API can't resolve 'cloudsql-proxy' hostname
    -> _get_or_create_oauth2_user() fails silently (returns None)
      -> /auth/check returns {authenticated: false}
        -> Frontend AuthProvider sees !user
          -> JS redirect to /oauth2/sign_in
            -> OAuth succeeds but same check fails again -> LOOP
```

### Why cookie_secure Was NOT the Issue
The OAuth2 cookie was working correctly the entire time. The `/oauth2/auth` endpoint returned 202 (authenticated) after login. The issue was downstream: the FastAPI app couldn't look up the user in PostgreSQL because the Cloud SQL Proxy sidecar was dead.

### Recommendations

1. **Immediate fix**: Restart Cloud SQL Proxy container:
   ```bash
   cd /opt/catscan && sudo docker-compose -f docker-compose.gcp.yml up -d cloudsql-proxy
   ```

2. **Resilience improvements** (code changes):
   - **`api/session_middleware.py`**: The `_get_or_create_oauth2_user()` method silently swallows DB errors and returns `None`, making `/auth/check` return "not authenticated" when the real problem is "database unreachable". This should return a 503 instead so the frontend can show "Service temporarily unavailable" rather than looping.
   - **`api/auth_oauth_proxy.py` line 148-151**: The `/auth/check` endpoint should distinguish between "not authenticated" and "auth system error" - return 503 or `{"authenticated": false, "error": "database_unavailable"}`.
   - **`dashboard/src/contexts/auth-context.tsx` line 127-129**: The blind redirect to `/oauth2/sign_in` when `!user` should have a retry/backoff mechanism and distinguish between "not logged in" vs "API error". Currently any API failure triggers an infinite redirect loop.

3. **Infrastructure improvements**:
   - Add a health check for Cloud SQL Proxy in the API's `/health` endpoint (currently passes even when DB is down)
   - Add `depends_on` with health check condition for cloudsql-proxy in docker-compose
   - Consider a deployment script that verifies all containers are running after deploy
   - Add monitoring/alerting for the cloudsql-proxy container status

4. **Auth check hardening** (prevents loop regardless of cause):
   - Add a `max_auth_redirects` cookie/counter in the frontend to break infinite loops
   - Show an error page after 3 failed auth attempts instead of redirecting

## Permanent Login Loop Fix — Implemented

All three recommendations from the root cause analysis above are now implemented in code.
This is a **permanent, structural fix** — not a band-aid. The loop can no longer happen regardless
of whether the cause is a dead Cloud SQL Proxy, a network partition, or any other DB failure.

### Changes Made (3 files)

#### 1. `api/session_middleware.py` — Stop swallowing DB errors
- `_get_or_create_oauth2_user()` now accepts `request` parameter
- On DB exception: sets `request.state.auth_error = "database_unavailable"` and
  `request.state.auth_error_detail` with the actual error message
- Still returns `None` (so the request continues), but the error is now **visible** to downstream handlers

#### 2. `api/auth_oauth_proxy.py` — `/auth/check` returns 503 on DB failure
- After checking for `request.state.user`, checks for `request.state.auth_error`
- If auth_error is set: returns **HTTP 503** with `{"authenticated": false, "error": "database_unavailable", "detail": "..."}`
- If no error and no user: returns the normal `{"authenticated": false, "user": null}` (200)
- This lets the frontend distinguish "not logged in" from "service broken"

#### 3. `dashboard/src/contexts/auth-context.tsx` — Loop breaker + error UI
- **New `authError` state**: tracks `"service_unavailable"` | `"network_error"` | `null`
- **503 detection**: `checkAuth()` now checks `response.status === 503` and sets `authError = "service_unavailable"`
- **Redirect counter**: uses `sessionStorage` to count redirects within a 30-second window
  - Max 2 redirects allowed before breaking the loop
  - Counter resets on successful auth or after 30s of no redirects
  - `resetRedirectCount()` called on successful auth and manual retry
- **Error/retry UI**: when `authError` is set and `!user`, shows a full-page error screen with:
  - Amber warning icon
  - "Service Temporarily Unavailable" or "Connection Error" heading
  - Explanation text
  - **Retry button** (resets counter, re-runs checkAuth)
  - Hint: "If this persists, check that all containers are running on the VM"
- **No redirect on error**: the `useEffect` that redirects to `/oauth2/sign_in` now returns early
  if `authError` is set — it only redirects when the user is genuinely not authenticated

### How It Prevents the Loop (all scenarios)

| Scenario | Old behavior | New behavior |
|----------|-------------|-------------|
| Cloud SQL Proxy dead | `/auth/check` → 200 `{authenticated:false}` → redirect → loop | `/auth/check` → 503 → error UI with Retry |
| API container can't reach DB | Same silent failure → loop | Same 503 → error UI |
| Network timeout on `/api/auth/check` | `catch` → `setUser(null)` → redirect → loop | `catch` → `authError = "network_error"` → error UI |
| OAuth2 Proxy misconfigured | Redirect → no X-Email → `{authenticated:false}` → redirect → loop | After 2 redirects in 30s → loop breaker → error UI |
| Genuinely not logged in | Redirect to `/oauth2/sign_in` (correct) | Same — single redirect (counter stays ≤2) |

### China Readiness Note

Google OAuth (`accounts.google.com`) is blocked in mainland China. The current architecture
requires OAuth2 Proxy → Google. For China access:

**Option A (recommended, no code change):** Users connect via VPN. This is the standard practice
for Google-dependent services in China.

**Option B (future, if needed):** Add `AUTH_MODE=header_trust` env var that makes the API trust
`X-Email` from any reverse proxy (not just OAuth2 Proxy). Pair with Cloudflare Access or
Authentik for non-Google SSO. This is a ~20-line change in `session_middleware.py` if needed.

The auth loop fix itself helps China users because: when Google OAuth is slow or times out
(common behind the GFW), the frontend now shows an error page with a Retry button instead of
spinning in an infinite redirect loop.
