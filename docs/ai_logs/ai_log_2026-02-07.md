## Gmail Token Refresh — How-To

### Why the token expired
The OAuth consent screen in GCP project `catscan-prod-202601` is likely in **Testing** mode.
In Testing mode, Google revokes refresh tokens after **7 days**. Last successful import was
Jan 29 — exactly ~7 days after the token was last refreshed (Jan 23).

The error: `invalid_grant: Token has been expired or revoked.`

### A. Fix root cause (one-time): Publish the OAuth consent screen

1. Go to: https://console.cloud.google.com/apis/credentials/consent?project=catscan-prod-202601
2. Check **Publishing status** — likely says "Testing"
3. Click **"PUBLISH APP"** and confirm
4. This makes refresh tokens last indefinitely (no more 7-day expiry)

### B. Re-authorize (get a new token)

The script `scripts/gmail_auth.py` handles this. It needs a browser for Google consent.

**Option 1 — From your local machine:**

```bash
# 1. Copy the client secret from VM1
mkdir -p ~/.catscan/credentials
gcloud compute scp \
  catscan-production-sg:/home/catscan/.catscan/credentials/gmail-oauth-client.json \
  ~/.catscan/credentials/gmail-oauth-client.json \
  --zone asia-southeast1-b --tunnel-through-iap

# 2. Install deps
pip install google-auth-oauthlib google-api-python-client

# 3. Run the auth script (opens browser for Google consent)
python scripts/gmail_auth.py

# 4. Copy new token back to VM1
gcloud compute scp \
  ~/.catscan/credentials/gmail-token.json \
  catscan-production-sg:/tmp/gmail-token.json \
  --zone asia-southeast1-b --tunnel-through-iap

# 5. Move into place with correct ownership
gcloud compute ssh catscan-production-sg \
  --zone asia-southeast1-b --tunnel-through-iap --command \
  "sudo cp /tmp/gmail-token.json /home/catscan/.catscan/credentials/gmail-token.json && \
   sudo chown 999:999 /home/catscan/.catscan/credentials/gmail-token.json"
```

**Option 2 — From VM1 via SSH port forwarding:**

```bash

# When it prints "open browser at http://localhost:XXXXX",
```

## Investigation: Missing Geo/Publisher Data for Billing ID 72245759413

### Root Cause: Two disconnected data pipelines

The platform has two separate import paths that store geo data in different tables:

| Pipeline | Raw Table | Feeds into | Has data for buyer 299038253? |
|---|---|---|---|
| **CSV import** (Gmail RTB Troubleshooting reports) | `rtb_daily` | `config_geo_daily`, `config_publisher_daily` | **YES** — 885K rows, but `country = NULL/''` in ALL rows |

**The home page geo chart works** because it sources from `home_geo_daily` ← `rtb_geo_daily` (API pipeline).
**The config detail Geo/Publisher tabs are empty** because they source from `config_geo_daily` ← `rtb_daily` (CSV pipeline), and the CSVs lack country/publisher columns.

### Why the CSVs lack country/publisher

The CSV report type detection (`importers/flexible_mapper.py:325-330`) classifies any CSV with a "Creative ID" column as `PERFORMANCE_DETAIL` and imports it to `rtb_daily` — **even if the Country column is missing**:

```python
if has_creative_id:
    missing = []
    if not has_day:
        missing.append("day")
    return ("performance_detail", "rtb_daily", missing)
    # Only validates "day" is present, NOT "country" or "publisher"
```

The column mapping (`unified_importer.py:339-343`) defaults missing columns to empty string:

```python
"country": get_value(row, mapping, "country", ""),      # → "" if not in CSV
"publisher_id": get_value(row, mapping, "publisher_id", ""),  # → "" if not in CSV
```

The precompute aggregation (`config_precompute.py:216,254`) explicitly filters these out:

```sql
-- config_geo_daily (line 216)
WHERE country IS NOT NULL AND country != ''

-- config_publisher_daily (line 254)
WHERE publisher_id IS NOT NULL AND publisher_id != ''
```

### Why Creative 102850 modal is empty

- **Country targeted**: `GET /creatives/102850/countries` queries `rtb_daily WHERE creative_id = '102850' AND country != ''` → 0 rows (same root cause)
- **Language detected**: `detected_language` field on the creative record is NULL — language analysis was never triggered for this creative
- **Performance data**: Spend stats endpoint returns 0 because it queries the same `rtb_daily` data

### Key files in the chain

| Step | File | Lines | What happens |
|---|---|---|---|
| CSV column detection | `importers/flexible_mapper.py` | 281-354 | Classifies as PERFORMANCE_DETAIL if has Creative ID, doesn't require Country |
| Column mapping defaults | `importers/unified_importer.py` | 339-343 | `country=""`, `publisher_id=""` when not in CSV |
| Value extraction | `importers/unified_importer.py` | 86-91 | `get_value()` returns default if column not mapped |
| INSERT to rtb_daily | `importers/unified_importer.py` | 287-296 | Inserts with empty strings |
| Geo precompute | `services/config_precompute.py` | 199-227 | `WHERE country != ''` excludes empty rows |

### The architectural gap

`config_geo_daily` and `config_publisher_daily` precompute exclusively from `rtb_daily` (CSV pipeline). They never consult `rtb_geo_daily` or `rtb_publisher_daily` (API pipeline). When a buyer's CSVs don't include geographic dimensions, the config detail page has zero geo/publisher data — even though the home page (sourced from the API pipeline) shows it correctly.

### Fix direction

The `config_geo_daily` precompute needs to also pull from `rtb_geo_daily` as a fallback/supplement when `rtb_daily.country` is empty for a given buyer. Similarly for publisher data. This is an architectural reconciliation between the two import pipelines — documented in `docs/efficiency-qps-reconciliation-plan-2026-02-07.md`.

## Root Cause Found: Gmail Import Downloading Old-Template CSVs

### Investigation

The previous commit (`957b97c`) removed buyer-level fallback from config breakdowns —
correct that account-wide data shouldn't masquerade as config-specific, but the real
question was: **why is billing-level geo data missing in the first place?**

#### Data pipeline trace

| Table | Billing 153322387893 rows | Country populated? |
|---|---|---|
| `rtb_daily` | 30 rows | **0** — all `country = ''` |
| `config_geo_daily` | 0 | N/A |
| `fact_delivery_daily` | 0 (billing scope) | N/A |
| `rtb_bidstream` (buyer-level) | 8,280 rows (buyer 299038253) | **Yes** — 15 countries |

All 885K rows for buyer 299038253 in `rtb_daily` have `country = ''`. Not a single row
with country data. Same for `publisher_id`. Only `creative_size` was populated.

#### CSV report comparison

Checked actual CSVs on VM1 (`/home/catscan/.catscan/imports/`):

| File | Header | Has Country? |
|---|---|---|
| `catscan-quality-299038253-*` (attachment-extracted) | `#Billing ID,Creative ID,Day,Hour,Creative size,Creative format,...` | **No** |
| `catscan-quality-6634662463-*` (attachment-extracted) | Same as above | **No** |
| `catscan-report-6574658621-*` (GCS URL download) | `#Billing ID,Creative ID,Country,Creative size,...` | **Yes** |
| `catscan-report-6634662463-*` (GCS URL download) | `#Billing ID,Creative ID,Creative size,...,Country,...` | **Yes** (some) |
| Manually downloaded from Gmail | `#Billing ID,Creative ID,Country,Creative size,...` | **Yes** |

**The scheduled report templates WERE updated** to include Country. But the import pipeline
was downloading the wrong version.

#### Root cause: attachment vs GCS URL priority

Google RTB report emails contain:
1. **Email attachment** — cached/pre-generated CSV using the **old** template (no Country)
2. **GCS download link in body** — freshly-generated CSV with **current** template (with Country)

`gmail_import.py` line 822-833:
```python
# First, try to extract CSV attachments (reports < 10MB)
downloaded_files = extract_attachments(service, message_id, payload)

# If no attachments, look for download URL (reports >= 10MB)
if not downloaded_files:
    url = extract_download_url(body)
    ...
```

For small reports (< 10MB), the attachment was found and used — **never reaching the GCS
download**. Buyer 6574658621's reports are large enough (> 10MB) that no attachment exists,
so GCS download kicked in and got the updated template.

### Fix (commit `661037b`)

Reversed download priority: try GCS URL first (latest template), fall back to attachment.

```python
# First, try GCS URL download (always has latest report template).
url = extract_download_url(body)
if url:
    try:
        downloaded_files = download_from_url(url, message_id, access_token, seat_id)
    except Exception as e:
        print(f"  GCS download failed ({e}), falling back to attachment", flush=True)

# Fall back to email attachment if GCS download unavailable or failed
if not downloaded_files:
    downloaded_files = extract_attachments(service, message_id, payload)
```
