## Fix Session: Startup Blockers and Response Model Regressions

### 1. Root Causes Confirmed (with file:line evidence)

#### RC-1: Migration startup blocker (fresh DB only)

`scripts/postgres_migrate.py:68` creates `schema_migrations` with only
`(version, applied_at)`. Migrations that reference `description`:
- `storage/postgres_migrations/027_schema_alignment.sql:207` — `INSERT INTO schema_migrations (version, applied_at, description)`
- `storage/postgres_migrations/039_ingestion_runs_extend.sql:12` — same pattern
- `storage/postgres_migrations/040_web_domain_lane.sql:30` — same pattern

On a fresh DB, `ensure_migrations_table()` runs first (no `description` column),
then migration 027 executes its SQL which INSERTs into `(version, applied_at, description)`
→ Postgres error: column "description" does not exist → migration marked failed →
`init_postgres_database()` raises `RuntimeError` → **API refuses to start**.

Existing production DBs are unaffected because 027 was applied before the DDL
diverged, so `description` already exists.

#### RC-2: `qps_current` type mismatch → 500 on `/settings/endpoints`

- `api/routers/settings/models.py:26` — `qps_current: Optional[int]`
- `storage/postgres_repositories/endpoints_repo.py:72` — `get_current_qps()` returns `float` from `SUM(current_qps)`
- `services/endpoints_service.py:26` — declared return type `-> float`

**Live proof (pre-fix):**
```
curl -H 'X-Email: cat-scan@rtb.cat' http://127.0.0.1:8000/settings/endpoints?buyer_id=6574658621
→ 500: "Input should be a valid integer, got a number with a fractional part
        [type=int_from_float, input_value=6299.495, input_type=float]"
```

#### RC-3: `/sizes` response contract mismatch → 500

- `api/routers/system.py:475` — `SizesResponse` model: `sizes: list[str]`
- `storage/postgres_store.py:301` — `get_available_sizes()` returns `list[dict]`
  with keys `canonical_size`, `size_category`, `count`
- `api/routers/system.py:520` — passes dict list directly to `SizesResponse(sizes=sizes)`

**Live proof (pre-fix):**
```
curl -H 'X-Email: cat-scan@rtb.cat' http://127.0.0.1:8000/sizes
→ 500: Internal Server Error
```

API startup log showed the same Pydantic error for this path:
```
Input should be a valid string [type=string_type,
  input_value={'canonical_size': 'Non-Standard', 'count': 1}, input_type=dict]
```

#### RC-4: Login page masks all errors as "Network error"

- `dashboard/src/app/login/page.tsx:47` — `response.json()` called before checking `response.ok`
- When nginx returns 502/504 HTML, `.json()` throws `SyntaxError`
- `catch (err)` at line 58 shows "Network error. Please try again." for all failures
- User cannot distinguish backend-down from wrong credentials

### 4. Remaining Risks and Follow-up

1. **Worker freeze root cause not conclusively identified** — Google API calls and
   Postgres both use `run_in_executor`, so the 2-hour unresponsive window (01:13–03:10
   UTC) has no confirmed trigger. Starlette `BaseHTTPMiddleware` is a known risk factor.
   **Follow-up:** Set `UVICORN_WORKERS=2` in VM2 `.env` to mitigate; monitor for recurrence.

2. **Startup Pydantic errors for `sizes`** — The `/sizes` endpoint fix prevents the
   500 at request time. However, if there is a startup cache-warming path that calls
   `get_sizes()` through the old code path, it may still log validation warnings
   during initialization. **Follow-up:** Trace any startup preload that calls
   `get_available_sizes()` and ensure it uses the same extraction pattern.

3. **Migration version key type mismatch** — `ensure_migrations_table()` and 001_init
   use `version TEXT`, but migrations 027/039/040 insert integer literals (`27`, `39`,
   `40`) while `apply_migration()` inserts string keys like `"027_schema_alignment"`.
   The `ON CONFLICT (version) DO NOTHING` prevents duplicates, but the same migration
   may appear twice with different version keys. **Follow-up:** Audit `schema_migrations`
   table for duplicate entries and consider normalizing version format.

4. **Dashboard rebuild required** — The login page fix is a frontend change. It will
   take effect only after the next `npm run build` / Docker image build via CI.
