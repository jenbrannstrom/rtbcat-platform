**WIN RATE >100% - ROOT CAUSE & FIX STATUS**

## Root Cause Identified

The win rate >100% bug was caused by the BigQuery aggregation script (`scripts/bq_aggregate_to_pg.py`) aggregating ALL rows from `rtb_bidstream` without filtering by `report_type`.

**Problem Data:**
- `rtb_bidstream` table contains multiple report types:
  - `rtb_bidstream_publisher` - has both `reached_queries` and `impressions` (correct data)
  - `rtb_bidstream_geo` - has `impressions` but `reached_queries = 0` (geo-level breakdown)
  - `NULL` report_type rows - cumulative/bad data

**Impact:**
- Aggregating all report_types sums impressions from geo breakdowns
- But reached_queries only exists in publisher rows
- Result: impressions inflated, reached_queries correct → win_rate = impressions/reached > 100%

**Verified in BigQuery:**
```sql
-- With filter (CORRECT):
SELECT SUM(reached_queries), SUM(impressions)
FROM rtb_bidstream
WHERE report_type = 'rtb_bidstream_publisher'
-- Result: 17,483,794 reached, 9,827,858 imps → 56.2% win rate

-- Without filter (BUG):
SELECT SUM(reached_queries), SUM(impressions) FROM rtb_bidstream
-- Result: 17,483,794 reached, 35,917,XXX imps → >100% win rate
```

**MIGRATION 028 FIX - SQLite vs Postgres Architecture Issue**

## Problem

Container was crashing on startup with:
```
RuntimeError: Database migration failed: 1 migration(s) failed
```

## Root Cause

Migration 028 (`028_add_creative_size_to_config_creative_daily.sql`) was placed in the **wrong location**:

| Item | Expected | Actual |
|------|----------|--------|
| Migration folder | `sql/postgres/` | `migrations/` (SQLite) |
| Target database | Postgres | SQLite |
| Syntax used | Postgres (`IF NOT EXISTS`) | SQLite doesn't support this |

The `migrations/` folder contains SQLite migrations (runner uses `sqlite3` module).
The `config_creative_daily` table is a **Postgres precompute table**, not SQLite.

## Architecture Clarification

This codebase has **two databases**:

| Database | Purpose | Migration Location |
|----------|---------|-------------------|
| **SQLite** | Local state, auth, configs | `migrations/*.sql` (via `migrations/runner.py`) |
| **Postgres** | Precompute/aggregation tables | `sql/postgres/` + `_ensure_tables()` in services |

The `config_*_daily` tables (size, geo, publisher, creative) are Postgres tables created by `services/config_precompute.py`:
- `_ensure_tables(conn)` function creates tables with correct schema
- Uses `pg_transaction_async()` to run against Postgres
- Already includes `creative_size TEXT` column in table definition

## Fix Applied

**Commit:** `3fa0717` - Remove migration 028: config_creative_daily is Postgres, not SQLite

Simply deleted the migration file since:
1. Table is Postgres, not SQLite
2. Column already defined in `_ensure_tables()` in `services/config_precompute.py`
3. No separate Postgres migration needed - service creates table on first run

## Additional Issue Found: Duplicate Migration Numbers
